Trust Workflow Layers

Every transaction flows through five trust layers. Each layer builds on the previous to create end-to-end accountability between buyers and factories.

1
Identity
USCC verification, legal rep, business license
➡
2
Capability
Live capacity, pricing rules, instant quotes
➡
3
Execution
Orders, milestones, photo evidence, QC inspection
➡
4
Payment
Escrow: deposit lock, production release, final release
➡
5
Dispute
Evidence-based resolution, refund or release
trust-layers.txt 
  Layer 1: IDENTITY — Who is this factory?
  verify_factory_identity  GET /factories/:id/verify-identity
    USCC lookup, legal rep, business license expiry, gsxt.gov.cn registry

  Layer 2: CAPABILITY — Can they build what I need?
  query_live_capacity      GET /capacity?category=...&qty=...
  get_instant_quote        GET /factories/:id/instant-quote?category=...&qty=...
  factory_performance      GET /factories/:id/performance
    Real-time capacity, tiered pricing, earned trust metrics

  Layer 3: EXECUTION — Is the order progressing?
  report_milestone         POST /orders/:id/milestones
  request_qc_inspection    POST /orders/:id/qc-request
    Photo-evidenced milestones, third-party QC (QIMA, SGS, Bureau Veritas)

  Layer 4: PAYMENT — Is money protected?
  lock_deposit             POST /orders/:id/escrow/lock
  confirm_receipt          POST /orders/:id/release-escrow
    30% deposit lock → auto-release on production → final release after QC pass

  Layer 5: DISPUTE — What if something goes wrong?
  raise_dispute            POST /orders/:id/dispute
    Evidence-based disputes, admin resolution (refund_full | refund_partial | rejected)

Quick Start: 3-Call Happy Path

From discovery to placed order in three API calls. Works with both REST and MCP.

1
search_factories
Find verified factories by category
➡
2
get_quote
Get binding price + lead time
➡
3
place_order
One POST, escrow-protected order
quickstart.sh 
# Step 1 — Search for PCB assembly factories
curl -s "$BASE/factories?category=pcb_assembly&verified_only=true" | jq
# Returns: { "factories": [{ "id": "sz-006", "name": "Nanshan IoT Systems", ... }], "count": 3 }

# Step 2 — Get a binding quote from the best match
QUOTE_ID=$(curl -s -X POST $BASE/quotes \
  -H "Content-Type: application/json" \
  -d '{"factory_id":"sz-006","product_description":"4-layer PCB 50x80mm","quantity":5000}' \
  | jq -r '.quote_id')
# Returns: { "quote_id": "q-...", "unit_price_usd": 4.20, "total_price_usd": 21000, "lead_time_days": 21 }

# Step 3 — Place the order (payment held in escrow)
ORDER_ID=$(curl -s -X POST $BASE/orders \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d "{\"quote_id\":\"$QUOTE_ID\",\"buyer_id\":\"buyer-001\"}" \
  | jq -r '.order_id')
# Returns: { "order_id": "ord-...", "escrow_status": "pending_deposit", "escrow_held": true }

# Step 4 — Lock 30% deposit
curl -s -X POST "$BASE/orders/$ORDER_ID/escrow/lock" \
  -H "Authorization: Bearer $TOKEN"
# Returns: { "deposit_locked": true, "deposit_amount_usd": 6300, "escrow_status": "deposit_held" }

# Step 5 — Track milestones as factory reports progress
curl -s "$BASE/orders/$ORDER_ID/milestones" \
  -H "Authorization: Bearer $TOKEN"
# Returns: { "milestones": [{ "milestone": "production_started", "photo_urls": [...] }] }

# Step 6 — Confirm receipt → release final escrow to factory
curl -s -X POST "$BASE/orders/$ORDER_ID/release-escrow"
# Returns: { "escrow_status": "final_released", "message": "Funds paid to factory" }

Escrow State Machine

Every order carries an escrow_status. The platform enforces valid transitions — invalid moves return 400.

escrow-states.txt 
  ┌─── Happy Path ──────────────────────────────────────────────────┐

  pending_deposit ───→ deposit_held ───→ production_released ───→ final_released ✓
      POST /escrow/lock      milestone: production_started       POST /release-escrow
      (buyer locks 30%)       (auto-transition)                    (requires qc_pass)

  ┌─── Dispute Paths ───────────────────────────────────────────────┐

  deposit_held ───→ disputed ───→ refunded ✓          (full/partial refund)
  production_released ───→ disputed ───→ final_released ✓  (dispute rejected)

  ┌─── Cancellation ────────────────────────────────────────────────┐

  pending_deposit ───→ refunded ✓   (cancelled before deposit)
  deposit_held ───→ refunded ✓      (refund before production)

Transition Table

From Status Allowed Transitions Trigger
pending_deposit deposit_held, refunded POST /orders/:id/escrow/lock, or cancel
deposit_held production_released, disputed, refunded Auto on production_started milestone, dispute, or refund
production_released final_released, disputed POST /orders/:id/release-escrow (requires qc_pass), or dispute
final_released TERMINAL Payment complete — no further transitions
disputed refunded, production_released, final_released POST /disputes/:id/resolve (admin)
refunded TERMINAL Funds returned — no further transitions
