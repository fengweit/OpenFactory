API Documentation

Interactive reference for the 5 trust layers — REST endpoints + MCP tool names for AI agent developers

Two interfaces, one platform: Every endpoint is available as both a REST API (for traditional integrations) and an MCP tool (for AI agents). Each section below shows both.

Authentication

OpenFactory supports two authentication methods: JWT tokens for user sessions and API keys for programmatic / AI-agent access.

JWT Token Flow

For buyers, factories, and admins. Tokens expire after 7 days.

# Register a buyer account
curl -X POST "$BASE_URL/auth/register" \
  -H "Content-Type: application/json" \
  -d '{"email":"buyer@example.com","password":"securepassword","role":"buyer"}'

# Response
{ "user_id": "usr-a1b2c3", "token": "eyJ..." }

# Login
curl -X POST "$BASE_URL/auth/login" \
  -H "Content-Type: application/json" \
  -d '{"email":"buyer@example.com","password":"securepassword"}'

# Response
{ "user_id": "usr-a1b2c3", "role": "buyer", "token": "eyJ..." }

API Key Flow

For AI agents and partner integrations. Keys are prefixed ofk_ and rate-limited per minute.

# Generate key (admin only)
curl -X POST "$BASE_URL/admin/api-keys" \
  -H "Authorization: Bearer $ADMIN_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"partner_name":"my-agent","permissions":["GET /factories*","POST /quotes"],"rate_limit_per_min":60}'

# Response
{ "key": "ofk_a1b2c3...", "id": "apk-..." }

# Use in requests
curl -H "Authorization: Bearer ofk_a1b2c3..." "$BASE_URL/factories"

MCP Server (no auth needed)

AI agents connecting via MCP use stdio transport — no auth tokens required.

# Claude Desktop config (claude_desktop_config.json)
{
  "mcpServers": {
    "openfactory": {
      "command": "npx",
      "args": ["ts-node", "src/mcp/server.ts"]
    }
  }
}
Auth on endpoint cards: Auth = requires Bearer token (JWT or API key). Public = no authentication required. tool_name = equivalent MCP tool for AI agents.

Base URL

# All endpoints are relative to your deployment origin
BASE_URL="https://your-openfactory-instance.com"

# In browser code, use window.location.origin
# Never hard-code localhost
Rate limits: 100 requests/minute per IP. Auth routes have stricter limits.

Error Handling

All errors return JSON with a consistent shape:

{
  "error": "Human-readable error message"
}
StatusMeaning
400Bad request — invalid input or missing required fields
401Unauthorized — missing or invalid Bearer token
403Forbidden — insufficient permissions
404Not found — resource doesn't exist
409Conflict — invalid state transition (e.g. escrow already locked)
422Unprocessable — invalid status transition
429Rate limited — 100 req/min exceeded

The 5 Trust Layers

OpenFactory builds buyer confidence through progressive trust. Each layer adds verifiable proof that reduces risk at every stage of cross-border manufacturing.

Progressive Trust Chain
1. Identity→ 2. Capability→ 3. Execution→ 4. Payment→ 5. Dispute→ Trust Earned
Layer 1: Identity Trust
Is this factory legally real? USCC verification, business license, legal representative — all cross-referenced with China's national registry.
Layer 2: Capability Trust
Can this factory actually make what I need? Real-time capacity data, instant pricing, and proven production capabilities.
Layer 3: Execution Trust
Is the factory delivering on its promises? Milestone tracking with photo proof, enforced ordering, and timestamped audit trails.
Layer 4: Payment Trust
Is my money safe? Milestone-gated escrow — funds only release when real production progress is verified with evidence.
Layer 5: Dispute Trust
What if something goes wrong? Structured disputes with evidence, escrow freezes, and platform mediation as a safety net.
Trust Earned
Each completed order increases the factory's composite trust score. New buyers benefit from the verified track record of previous transactions.

Layer 1 — Identity Trust

1
Is this factory legally real?

Every factory on OpenFactory has a verified Unified Social Credit Code (USCC) — China's business identity number. We validate the checksum, cross-reference with gsxt.gov.cn, and expose the legal representative and license expiry.

GET /factories/:id/verify-identity Public verify_factory_identity

Identity trust data for buyer due diligence. Returns USCC code, legal representative, license expiry, and a direct link to China's national enterprise registry.

REST Endpoint
GET /factories/:id/verify-identity
MCP Tool
verify_factory_identity({ factory_id })
Example

Layer 3 — Execution Trust

3
Is the factory delivering on its promises?

Production milestones with photo evidence create an immutable audit trail. Milestone ordering is enforced — you can't report "shipped" without first passing QC.

Milestone Ordering
material_received→ production_started→ qc_in_progress→ qc_pass→ ready_for_shipment→ shipped

POST /orders/:id/milestones Auth report_milestone

Factory reports a production milestone. Enforces ordering (e.g. production_started requires material_received).

REST Endpoint
POST /orders/:id/milestones
MCP Tool
report_milestone({ order_id, milestone, photo_urls, note })
Request Body
FieldTypeRequiredDescription
milestonestringYesmaterial_received | production_started | qc_in_progress | qc_pass | qc_fail | ready_for_shipment | shipped
photo_urlsstring[]NoArray of photo URLs documenting this milestone
notestringNoOptional note (e.g. QC results, tracking number)
Example
curl -X POST "$BASE_URL/orders/ord-e5f6g7h8/milestones" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "milestone": "production_started",
    "photo_urls": ["https://photos.example.com/pcb-line-start.jpg"],
    "note": "Materials sourced, PCB fabrication started"
  }'
Response 201
{
  "id": 1,
  "order_id": "ord-e5f6g7h8",
  "milestone": "production_started",
  "reported_by": "factory-user-001",
  "note": "Materials sourced, PCB fabrication started",
  "created_at": "2026-03-28T08:00:00.000Z",
  "photo_upload_url": "/orders/ord-e5f6g7h8/milestones/1/photos"
}

Layer 4 — Payment Trust

4
Is my money safe?

Milestone-gated escrow means funds only move when real progress is verified. The buyer locks a 30% deposit, and the remaining 70% releases in stages as production milestones are met with photo evidence.

Escrow State Machine
pending_deposit→ deposit_held→ production_released→ qc_released→ final_released
any state→ disputed→ resolved
TransitionTriggerAmount
pending_deposit → deposit_heldBuyer locks deposit30% of total
deposit_held → production_releasedproduction_started milestone + photo20% of total
production_released → qc_releasedqc_pass milestone + photo20% of total
qc_released → final_releasedBuyer confirms receipt30% of total
any → disputedDispute raised by any partyFunds frozen

POST /orders/:id/escrow/lock Auth lock_deposit

Lock 30% deposit on an order. Transitions escrow from pending_deposit to deposit_held.

REST Endpoint
POST /orders/:id/escrow/lock
MCP Tool
lock_deposit({ order_id, amount_usd, note })
Example
curl -X POST "$BASE_URL/orders/ord-e5f6g7h8/escrow/lock" \
  -H "Authorization: Bearer $TOKEN"
Response
{
  "order_id": "ord-e5f6g7h8",
  "deposit_locked": true,
  "deposit_amount_usd": 3570.00,
  "remaining_usd": 8330.00,
  "escrow_status": "deposit_held",
  "message": "Deposit locked. Factory can now begin production."
}
Idempotency: Returns 409 if escrow is already past pending_deposit state.

POST /orders/:id/escrow/release Auth confirm_receipt

Milestone-based escrow release. Validates milestone completion and photo evidence before allowing each transition.

REST Endpoint
POST /orders/:id/escrow/release
MCP Tool
confirm_receipt({ order_id }) for final release
MCP tools for escrow: Use lock_deposit to lock the initial 30%, check_escrow_status to view the full escrow audit trail, and confirm_receipt for the final release after delivery.
Request Body
FieldTypeRequiredDescription
milestonestringYesproduction_started | qc_pass | shipped
verified_bystringNoWho verified the milestone (e.g. QC inspector ID)
Example
curl -X POST "$BASE_URL/orders/ord-e5f6g7h8/escrow/release" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"milestone": "production_started"}'
Response

Layer 5 — Dispute Trust

5
What if something goes wrong?

When issues arise, disputes freeze escrow immediately. Both parties submit evidence, and the platform mediates. The structured process ensures no funds move until resolution.

POST /orders/:id/dispute Auth raise_dispute

Raise a dispute on an order. Transitions escrow to disputed state.

REST Endpoint
POST /orders/:id/dispute
MCP Tool
raise_dispute({ order_id, raised_by, reason, evidence_urls })
Request Body
FieldTypeRequiredDescription
raised_bystringYesbuyer | factory | platform
reasonstringYesReason for the dispute
evidence_urlsstring[]NoArray of evidence URLs (photos, documents)
Example
curl -X POST "$BASE_URL/orders/ord-e5f6g7h8/dispute" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "raised_by": "buyer",
    "reason": "Received units do not match approved sample",
    "evidence_urls": ["https://evidence.example.com/photo1.jpg"]
  }'
Response 201
{
  "id": 1,
  "order_id": "ord-e5f6g7h8",
  "raised_by": "buyer",
  "reason": "Received units do not match approved sample",
  "evidence_urls": ["https://evidence.example.com/photo1.jpg"],
  "status": "open",
  "escrow_status": "disputed",
  "created_at": "2026-04-12T09:00:00.000Z"
}
Escrow freeze: Raising a dispute immediately freezes all escrow transitions. No funds can be released until the dispute is resolved.

Typical Agent Flow

Here's how an AI agent walks through all 5 trust layers in a single procurement workflow:

End-to-End MCP Tool Chain
search_factories→ verify_factory_identity→ get_instant_quote→ place_order→ lock_deposit→ report_milestone→ confirm_receipt 
// Step 1 — Identity Trust: verify the factory is real
const identity = await mcp.call("verify_factory_identity", { factory_id: "sz-006" });
// Check: identity.verified === true, identity.uscc_metadata.checksum_valid === true

// Step 2 — Capability Trust: get instant pricing
const quote = await mcp.call("get_instant_quote", {
  factory_id: "sz-006", category: "pcb_assembly", quantity: 5000
});

// Step 3 — Execution Trust: track milestones
const ms = await mcp.call("report_milestone", {
  order_id: "ord-e5f6g7h8", milestone: "production_started",
  photo_urls: ["https://photos.example.com/line-start.jpg"]
});

// Step 4 — Payment Trust: release escrow on milestone
const escrow = await mcp.call("check_escrow_status", { order_id: "ord-e5f6g7h8" });

// Step 5 — Dispute Trust (if needed)
const dispute = await mcp.call("raise_dispute", {
  order_id: "ord-e5f6g7h8", raised_by: "buyer",
  reason: "Units don't match sample"
});
