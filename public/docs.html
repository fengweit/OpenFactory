7 Milestone Types & Dependency Ordering

Milestones must be reported in dependency order. A milestone cannot be created until all its prerequisites exist.

#TypePrerequisitesDescriptionEscrow Effect
1material_receivednoneMaterials arrive at factory—
2production_startedmaterial_receivedManufacturing beginsdeposit_held → production_released

🔑 Authentication

Most endpoints require a JWT token or API key. Obtain a JWT via POST /auth/login, then pass it as a Bearer token. Alternatively, use an API key via the x-api-key header.

// Bearer token Authorization: Bearer <jwt_token> // API key x-api-key: <your_api_key>

📋 Milestones

Report production milestones with dependency enforcement. Each milestone type has prerequisites that must be met.

POST /orders/:id/milestones Report a milestone ▶
Request Body
FieldTypeDescription
typeenumREQOne of: material_received, production_started, qc_in_progress, qc_pass, qc_fail, ready_for_shipment, shipped
notestringOPTFree-text note about this milestone
photo_urlsstring[]OPTInline photo URLs as evidence
Dependency Rules

The server enforces ordering. For example, production_started cannot be reported until material_received exists. Attempting to skip a step returns 400 with a message listing missing prerequisites.

Escrow Auto-Trigger

When production_started is reported and escrow status is deposit_held, the system automatically transitions escrow to production_released (70% of funds released to factory).

Example Request
{ "type": "production_started", "note": "Assembly line A running. Batch #2024-0312.", "photo_urls": ["/uploads/milestones/ord-001/ms-002/line-a.jpg"] }
Response 201
{ "id": 2, "order_id": "ord-001", "type": "production_started", "note": "Assembly line A running. Batch #2024-0312.", "photo_urls": ["/uploads/milestones/ord-001/ms-002/line-a.jpg"], "reported_by": "factory", "created_at": "2025-03-12T09:15:00Z", "escrow_triggered": "production_released" }
Error 400 — Missing Prerequisites
{ "error": "Missing prerequisite milestones: material_received" }

💰 Escrow

Milestone-based escrow protects both buyers and factories. Funds are locked at order creation and released as milestones are verified.

Escrow State Machine

// Valid state transitions pending_deposit → deposit_held | refunded deposit_held → production_released | disputed | refunded production_released → final_released | disputed final_released → (terminal) disputed → refunded | production_released | final_released refunded → (terminal)
POST /orders/:id/escrow/lock Lock buyer deposit ▶
Description

Locks 30% of the order total as a deposit. Transitions escrow from pending_deposit to deposit_held. Called by the buyer after order confirmation.

Request Body
FieldTypeDescription
amount_usdnumberREQDeposit amount in USD (30% of order total)
Example Request
{ "amount_usd": 1500 }
Response 200
{ "order_id": "ord-001", "escrow_status": "deposit_held", "amount_locked_usd": 1500, "event": { "from": "pending_deposit", "to": "deposit_held", "trigger": "buyer_lock", "timestamp": "2025-03-10T14:00:00Z" } }
POST /orders/:id/escrow/release Release escrow funds ▶
Description

Releases escrow funds based on milestone completion. Transitions escrow to final_released after buyer confirms delivery. Photo proof must exist for key milestones (production_started, qc_pass, or shipped).

Request Body
FieldTypeDescription
milestone_typestringREQMilestone that triggers release
notestringOPTRelease note
Example Request
{ "milestone_type": "shipped", "note": "Buyer confirmed delivery. Releasing final payment." }
Response 200
{ "order_id": "ord-001", "escrow_status": "final_released", "event": { "from": "production_released", "to": "final_released", "trigger": "milestone_release", "amount_usd": 3500, "timestamp": "2025-03-20T16:45:00Z" } }
